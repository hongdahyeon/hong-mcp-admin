# Plan 01.

# 프론트엔드 기본 구조 설계 및 적용

React (Vite) + TypeScript 설정을 점검하고, 프로젝트 규모에 맞는 체계적인 폴더 구조와 **라우팅(React Router)** 기반의 아키텍처를 구축합니다.

## 제안 사항 및 설계 계획

1. **TypeScript 전환**:
   - `typescript`, `tslib`, `@types/node` 등 필수 패키지 설치
   - `tsconfig.json` 및 `tsconfig.app.json`, `tsconfig.node.json` 설정 파일 생성
   - `src/vite-env.d.ts` 파일 추가
   - 기존 `.jsx` 파일을 `.tsx`로 변환 (`App.tsx`, `main.tsx`)
2. **폴더 구조 자동화**: `src/` 하위에 다음과 같은 폴더를 생성합니다:
   - `api/`: API 호출 로직 (Axios 인스턴스 등)
   - `components/`: 재사용 가능한 UI 컴포넌트 (`common/`, `layout/`)
   - `hooks/`: 커스텀 Hook
   - `pages/`: 라우팅되는 페이지 컴포넌트
   - `routes/`: 라우터 설정 및 경로 매핑 (URL ↔ Component)
   - `store/`: 전역 상태 관리 (Zustand/Redux 등)
   - `types/`: 공통 TypeScript 인터페이스 및 타입 정의
   - `utils/`: 유틸리티 함수 및 상수

## 검증 계획

### 자동화 테스트

- `npm run build`를 통한 TypeScript 타입 체크 및 빌드 성공 여 확인

### 수동 검증

- `npm run dev`를 통한 로컬 개발 환경에서의 정상 실행 확인

# Plan 08.

# 전역 예외 처리 최적화 (ErrorResponse / HongErrorCode 연동)

백엔드에서 발생하는 `HongException`이 `HongErrorCode`를 통해 구조화된 `ErrorResponse`를 반환하도록 변경됨에 따라, 프론트엔드 Axios 인터셉터를 이에 맞춰 최적화합니다.

## 제안 사항 및 설계 계획

1. **상세 에러 구조 대응**:
   - `ErrorResponse` 구조(`code`, `message`, `status`)에 맞춰 응답 데이터를 파싱합니다.
   - 단순히 메시지만 보여주는 것이 아니라, 필요 시 에러 코드(`code`)를 활용한 분기 처리가 가능하도록 기반을 마련합니다.

2. **Axios Response Interceptor 고도화**:
   - `data.message`가 존재할 경우 최우선적으로 사용자에게 노출합니다.
   - **401 에러 세분화**: 단순히 `status === 401`인 경우 세션 만료로 처리하지 않고, `code`가 `A001`(Unauthorized), `A002`(Invalid Token) 등 인증 시스템 관련 에러인 경우에만 로그아웃 처리를 수행합니다. `U002`(Password Not Match)와 같은 비즈니스 예외는 메시지만 노출합니다.
   - 에러 코드와 함께 로그를 남겨 디버깅 편의성을 향상시킴으로써 디버깅 편의성을 향상시킵니다.

## 검증 계획

### 수동 검증

- 존재하지 않는 계정(`USER_NOT_FOUND`) 또는 비밀번호 불일치(`PASSWORD_NOT_MATCH`) 시 백엔드에서 내려주는 각기 다른 `message`가 정확히 노출되는지 확인합니다.
- 중복된 이메일/이름 가입 시도 시 `BAD_REQUEST(400)` 상태와 함께 구체적인 에러 메시지가 표시되는지 확인합니다.

# Plan 10.

# 레이아웃 고도화 및 테마 시스템 구축

사용자 경험 향상을 위해 전역 다크 모드 기능을 도입하고, 넓은 화면에서도 안정적인 레이아웃을 제공하며, 로그인 편의성을 위해 아이디 기억 기능을 구현합니다.

## 제안 사항 및 설계 계획

1. **전역 테마 시스템 (`ThemeContext`)**:
   - `React Context API`를 활용하여 `ThemeProvider` 구현.
   - `localStorage`와 연동하여 테마 설정 유지 및 브라우저 기본 설정(Media Query) 대응.
   - Tailwind CSS v4 `@variant dark` 설정을 통한 클래스 기반 다크 모드 지원.

2. **레이아웃 및 반응형 최적화**:
   - 메인 컨텐츠 영역에 `max-w-[1440px]` 및 `mx-auto`를 적용하여 광활한 화면에서도 가독성 확보.
   - 모든 주요 컴포넌트(Header, Footer, Home, Login, Signup)에 다크 모드 스타일(`dark:`) 적용.

3. **로그인 편의 기능**:
   - `localStorage` 키 `HONG_CRAFT_DAY`를 활용한 아이디Persistence 구현.
   - 체크박스 상태에 따른 아이디 저장/삭제 로직 및 자동 완성 기능.

## 검증 계획

### 수동 검증

- 헤더의 테마 토글 버튼을 통해 다크/라이트 모드가 즉각적으로 모든 페이지에 반영되는지 확인.
- 로그인 페이지에서 '아이디 기억하기' 체크 후 로그인 시, 재방문 시 아이디가 자동 입력되는지 확인.
- 2560px 이상의 고해상도 모니터에서 레이아웃이 좌우로 과하게 퍼지지 않고 중앙에 안정적으로 배치되는지 확인.

# 토큰 재발급 (Token Reissue) 플로우 구현

Access Token이 만료되었을 때 (`A005` 에러), 사용자가 로그아웃되지 않고 Refresh Token을 통해 새로운 Access Token을 발급받아 기존 요청을 자동으로 재시도하는 'Silent Refresh' 로직을 심플하게 구현합니다.

## 제안 사항 및 설계 계획

1. **`authService.reissue` 메서드 활용**:
   - `/api/auth/reissue` 엔드포인트에 `refreshToken`을 담아 요청.
   - `A005` 에러 발생 시에만 동작하도록 단순화합니다.

2. **Axios Response Interceptor 단순화**:
   - 응답 데이터의 `code` 값이 `A005`인 경우 즉시 `reissue`를 시도합니다.
   - 재발급 시도 중 다시 `A005`가 발생하거나 실패하면 즉시 세션 만료 알림을 띄우고 로그인 페이지로 이동합니다.
   - 그 외의 모든 에러는 `ErrorResponse`의 `message`를 그대로 활용하여 경고창을 띄웁니다.

3. **중복 방지 로직 유지**:
   - 최소한의 플래그(`isRefreshing`)를 활용하여 찰나의 순간에 중복 재발급 요청이 나가는 것을 방지합니다.

## 검증 계획

### 수동 검증

- Access Token을 고의로 만료시킨 후 페이지를 새로고침하거나 API를 호출했을 때, 자동으로 토큰이 갱신되고 데이터가 정상적으로 불러와지는지 확인.
- `localStorage`의 `AUTH_DATA` 내 토큰 값이 변경되는지 확인.
- Refresh Token까지 만료되었을 때 정상적으로 로그인 페이지로 리다이렉트되는지 확인.
